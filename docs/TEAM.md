
## 各组详细职责

### 🔧 开发1组（架构）

**定位**：地基，确保所有人能在同一个框架下协作

| 职责 | 具体内容 |
|------|---------|
| 核心框架维护 | `agents/base.py` · `agents/signal.py` 的修改与发布 |
| Agent 注册机制 | 新增 Agent 时的目录规范、注册流程 |
| Git 工作流 | develop 分支策略、PR 模板、冲突处理 |
| CI/CD | 测试用例、自动化检查（未来） |
| 调试工具 | `debug_ui/app.py` 的迭代维护 |

**负责人**：@monsignorc  
**分支权限**：`main` 保护规则制定者

---

### 🔧 开发2组（功能）

**定位**：让 Agents 的决策能被仲裁、被理解

| 职责 | 具体内容 |
|------|---------|
| 仲裁层 | `arbitration/engine.py` — 信号博弈、趋势一致性检查 |
| 认知层 | `agents/cognition/` — 推理链生成、认知图谱 |
| 主入口 | `main.py` — 多 Agent 调度、流程编排 |
| 信号融合 | 多 Agent 信号的加权、去重、冲突解决 |

---

### 🔧 开发3组（数据）

**定位**：让所有 Agent 不用担心数据从哪里来

| 职责 | 具体内容 |
|------|---------|
| 统一数据接入层 | 封装东方财富 / Tushare / 另类数据源 |
| API 封装 | 统一的 `fetch_*()` 接口，Agent 只需调用 |
| 本地缓存 | 避免重复请求，降低 API 限流风险 |
| 数据质量校验 | 缺失值处理、异常值标记、数据源可信度标注 |

**输出**：`data/` 目录 + 各 Agent 中的 `_fetch_*()` 方法

---

### 📊 专家1组（财务）

**定位**：财报质量的终极裁判

| 职责 | 具体内容 |
|------|---------|
| 七步验证链 | 利润现金比 → 需求真实性 → 合同负债 → 产能扩张 → 资本开支 → 净债务 → 财务费用侵蚀 |
| Skill 维护 | `skills/financial_report_analysis/SKILL.md` 的持续迭代 |
| 数据溯源规则 | A/B/C 类数据可信度标注 |
| 降级策略 | 无 LLM 时的本地规则引擎 |

**对应 Agent**：`agents/research/financial_report/agent.py`  
**对应 Skill**：`skills/financial_report_analysis/SKILL.md`

---

### 📈 专家2组（指标）

|------|---------|

## 跨组协作规范

### Agent 开发流程

```
专家组的 Agent 开发：
  1. 专家组分头实现 analyze()
  2. 分析逻辑沉淀到 skills/（Skill 与 Agent 分离）
  3. 开发1组 Review 代码规范
  4. 综合组跟踪进度

数据需求：
  1. 专家组提出数据需求 → Issue 标签 data-request
  2. 开发3组认领 → 实现数据接入
  3. 专家组验收数据质量
```

### Skill 共享机制

```
同一份 Skill 可被多个 Agent 调用：

financial_report_analysis  ← 财务专家Agent（A股）
                          ← 港股财报Agent（港股）
                          ← 财报对比Agent（跨期对比）

开发3组负责维护数据接入，
专家组负责维护 Skill 分析逻辑，
双方解耦，各自迭代。
```
